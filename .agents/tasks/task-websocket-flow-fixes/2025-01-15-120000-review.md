# Ref-based closure fix for WebSocket streaming handlers

The Chat component's WebSocket event handlers (`handleStreamEvent`, `handleComplete`, `handleError`) were registered once during `connectWebSocket` but defined as plain functions that closed over the initial render's state values. This meant `streamingText` and `thinkingText` were always empty strings inside those handlers, causing `handleComplete` to silently discard accumulated content and leave `isLoading` stuck on `true`—the "hang" bug. The fix introduces `streamingTextRef` and `thinkingTextRef` that shadow the state values, and wraps handlers in `useCallback([], ...)` so they read from refs instead of stale closure captures. A secondary fix rewrites the initial-query auto-submit to call `sendQuery` directly instead of routing through `handleSendMessage` (which itself suffered from the same stale-closure problem on `inputText`).

**Watch for:** The `connectWebSocket` function still closes over the `useCallback` handler references from the first render, which works only because the callbacks have empty dependency arrays and never change identity. If a future change adds dependencies to those callbacks, the registered listeners will go stale again (confirmed). Additionally, there is duplicated "send" logic between `handleSendMessage` and the initial-query `useEffect` that could drift (confirmed).

## High-level view

The core fix uses mutable refs to mirror streaming state so that event handlers registered once can always read current values. The handlers are wrapped in `useCallback` with empty deps to guarantee stable identity.

The initial-query auto-submit was rewritten to bypass `handleSendMessage` entirely. Instead of setting `inputText` and calling a function that reads it (a guaranteed stale read in the same tick), the effect duplicates the send logic inline: construct the message, append it, call `sendQuery`. This eliminates the race but introduces a second code path for sending messages that must be kept in sync manually.

`connectWebSocket` registers handlers via `wsService.on(...)` referencing `handleStreamEvent`, `handleComplete`, etc. These identities are stable (empty-dep `useCallback`), but if a future refactor adds dependencies to any handler, the reconnection path (visibility/online listeners) will silently register stale callbacks.

## Handler stability on reconnect

`connectWebSocket` is not wrapped in `useCallback` and is re-created on every render. It's called from a mount `useEffect` and from visibility/online event listeners in a separate `useEffect` (deps: `[sessionId, user]`). Inside `connectWebSocket`, the handler references (`handleStreamEvent`, etc.) are captured at call time. Because these are `useCallback` with `[]` deps, their identity never changes, so the captured references remain valid today. If a future refactor adds a dependency to any handler's `useCallback`, this chain breaks silently—reconnection listeners will register a stale handler and new stream events will be processed by the old closure.

A more robust pattern would store the handlers in refs (e.g., `handleStreamEventRef.current = handleStreamEvent`) and have `connectWebSocket` always read from the ref, decoupling registration from identity stability.

## Duplicated send path

`handleSendMessage` (lines 209–237) constructs a user `Message`, appends it, resets refs and state, then calls `wsServiceRef.current.sendQuery(...)`. The initial-query `useEffect` (lines 241–264) does the exact same sequence inline. The only behavioral difference is that `handleSendMessage` reads from `inputText` while the effect uses `initialQuery` from location state.

If either path is later modified (e.g., adding optimistic message IDs, telemetry, rate limiting), the other will not get the change. A shared `sendMessage(text: string)` function that both paths call would eliminate this risk.

## Silent completion with no content

`handleComplete` only appends a message to the conversation if `currentStreaming || currentThinking` is truthy. If the server emits a `complete` event without ever having sent `data` or `current_tool_use` events (edge case: empty agent response, backend timeout that resolves without output), the user sees the loading spinner disappear with no message added. There's no "empty response" placeholder or user-visible indication that the exchange completed. This is a pre-existing gap preserved by the refactor—noting it because the rewrite was a natural place to address it.

## Initial-query dependency array

The addition of `user` to the `useEffect` dependency array (absent in the original) ensures `user?.sub` is available for the `sendQuery` call. Without it, the effect could fire before auth state loads, sending `undefined` as the user ID.
